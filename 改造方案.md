# 项目经历

## 智能单车客服系统

**项目介绍**：智能单车客服系统通过 AI Agent 解决共享单车用户服务痛点。整合知识库、对话、投诉处理三大核心能力，实现用户问题自动应答与复杂投诉智能分析的一体化服务，致力于降低客服人力成本，提升用户满意度。

**技术栈**：SpringBoot + SpringAI-Alibaba + RAG + ReAct + Plan-Execute-Replan + Multi-Agent + MCP

**核心职责**：

- **设计AI Agent 架构**：基于 SpringAI-Alibaba 框架设计实现 Knowledge Index Agent、Chat ReAct Agent 和 Plan-Execute-Replan Agent，通过图编排实现模块化的 Agent 工作流。引入 MCP 协议集成工单查询工具、投诉历史查询工具、订单数据查询工具、联网查询等，实现外部系统接口的标准化插拔调用；

- **设计 RAG 知识库系统**：设计了文档向量化存储和检索方案与文档增量更新机制，支持客服知识库的智能检索增强生成；定制化调优文档分块策略与检索 TopK 参数，使知识库检索准确率稳定在 85%+；

- **开发对话功能**：基于 ReAct 模式开发对话Agent，实现开锁问题、计费争议、账户咨询等多场景业务处理与工单预处理。设计分层上下文记忆机制以突破大模型长对话上下文窗口限制，并编写结构化 Prompt 约束模型输出以减少幻觉。基于 SSE技术实现AI应答流式输出，解决大模型响应延迟问题；

- **开发智能投诉分析功能**：基于 Plan-Execute-Replan 模式开发智能投诉处理Agent。设计多 Agent（Planner、Executor、Replanner）协作机制，实现根据投诉工单 → 检索知识库 → 规划处理步骤 → 调用工具查询历史记录 → 分析投诉原因 → 生成处理建议的完整业务流程，应对复杂投诉场景。

## 三、多 Agent 场景设计（核心决策）

### 原 AIOps 场景 → 新客服投诉处理场景

原系统的多 Agent 架构（Planner → Executor → Replanner）非常适合**复杂投诉处理**场景，映射关系如下：

| 原 AIOps 概念 | 新客服概念 | 说明 |
|--------------|-----------|------|
| 告警分析报告 | 投诉处理报告 | 输出格式相似，内容换为客服场景 |
| Prometheus 告警 | 用户投诉工单 | 查询当前待处理的投诉/故障工单 |
| 日志查询 | 投诉历史记录查询 | 查询用户历史投诉和处理记录 |
| 内部文档检索 | 客服知识库检索 | 查询处理规范、赔偿标准、FAQ |
| 根因分析 | 投诉原因分析 | 分析投诉根本原因 |
| 处理方案 | 解决方案推荐 | 给出具体的客服处理建议 |

### 新多 Agent 工作流

```
用户触发投诉处理 → POST /api/service_agent
  → SupervisorAgent 调度：
    → PlannerAgent: 读取投诉工单，制定处理计划
    → ExecutorAgent: 查询投诉历史、检索知识库、分析原因
    → PlannerAgent: 根据反馈再规划，直到 decision=FINISH
  → 输出《投诉处理分析报告》（Markdown 格式，SSE 流式）
```

**适用的典型场景举例：**
- 用户反映单车无法解锁，系统自动分析是设备故障还是账户问题
- 用户投诉被多扣费，系统自动查询订单记录并给出赔偿建议
- 批量投诉同一区域单车故障，系统自动汇总并生成运维派单建议
